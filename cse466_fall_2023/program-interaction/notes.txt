--- Loading the binary ---

reading the interpreter of an elf:

	readelf -a /bin/cat | grep interpret --> returns the interpreter aka loader

it can be temporarily overridden:

	/pathto/someloader /yourbinary

or changed permanently:

	patchelf --set-interpreter /pathto/someloader



each linux process has a virtual memory space that contains:

	- the binary
	- libraries
	- heap (for dynamically allocated memory)
	- stack (for function local variables)
	- memory specifically mapped by the program
	- helper regions
	- kernel code unaccessible to the process

virtual mem vs physical

	virtual is assigned to a specific process 
	physical is shared

	see /proc/self/maps



standard C library

	libc.so is linked by most processes

	contains things like:
		- printf
		- scanf
		- socket
		- atoi
		- malloc
		- free 
		etc
------

--- Initializing the binary ---

constructors:

	each binary can specify constructors which are functions that run before the program is launched

	you can also create your own:

		__ attribute__((constructor)) void test() {

			puts("test");
		}

------


--- The binary launches ---

	normal ELFs call __libc_start_main() in libc which calls the programs main() function

	the process gets all input from 
		
		- loaded objects (binaries/libraries)
		- command line args in argv
		- environment from envp

		int main(int argc, void **argv, void **envp);

	interacting with the environment:

		usually this is done via syscalls (ex `man 2 open`)
		syscalls can be traced via strace

	signals:

	typical signal combos:

		fork -> execve -> wait
		open -> read -> write

	syscalls are a way for a program to call to the OS; signals are a way for the OS to "signal" to the program

	signals pause process execution and invoke a handler
	a handler is a function that takes the signal number as its one argument
	without a handler for the signal, the default action is taken (usually kill)

	see `man 7 signal` / `kill -l`
------

--- Process termination ---

	process either receives unhandled signal OR calls exit() syscall

	after termination, they will remain in a zombie state until they are wait() by their parent

	when that happens, their exit code is returned to the parent and then the process is freed
------

extra notes:

	when using subprocess.Popen or pwn.process, make sure to wait() for the children otherwise the parent will
	most likely just terminate
